# GWT 標的證據平台 — 使用者介面產品規格（兩入口）

**文件類型：** Product / UX 規格 · **作者：** PM (Claude Science) · **日期：** 2026-07-08
**產品定位：** 一套資料骨幹、兩個使用情境入口。研究者找**藥物開發證據**；臨床醫師用**病人自身資料辨識風險**。兩入口同等重要，共用同一證據引擎，但**任務、詞彙、責任邊界完全不同**。

> **貫穿設計原則（延續既有 decisions）**
> 1. **低認知負荷** — 每個畫面只回答一個問題；預設收合、漸進揭露。
> 2. **一圖一結論** — 每張圖的標題就是它的 takeaway（沿用 figure-style §5.1）。
> 3. **證據可追溯** — 每個數字都能點回原始來源（Open Targets / gnomAD / ClinicalTrials / Reactome / STRING / PubMed / GTEx）與時間戳。
> 4. **Demo 邊界明示** — 非臨床決策工具；病人層一律標「研究用假設，非處方」。
> 5. **誠實揭露限制** — 排名的泛效應污染、CRISPRi 限制等 caveat 主動顯示，不藏。

**設計要點：** 兩入口不是兩個 app，是**同一資料的兩種讀法**。研究者由「基因/標的」進入；醫師由「病人/疾病」進入。中間的證據引擎完全共用——這保證兩邊看到的證據一致、不會各說各話。

## 1. ENTRY A — 研究者：Target Evidence Explorer

**核心任務：** 「哪些基因值得投入藥物開發？證據多強？可成藥嗎？安全嗎？」

### 1.1 畫面流（三層漸進）

```
[L1 排名總覽] ──點標的──▶ [L2 標的證據卡] ──點某證據──▶ [L3 來源鑽取]
   Rank Board          Target Dossier            Source Drill-down
```

### L1 — Rank Board（排名總覽，落地首屏）

**回答的問題：** 「在我的條件下，最值得看的標的排名。」

| 區塊 | 內容 | 互動 |
|---|---|---|
| 主圖 | **Lollipop 排名圖**（R1，封面級）— 標的依所選指標排序，三條件(Rest/Stim8hr/Stim48hr)三點 | 點任一標的 → L2 |
| **排名指標切換**（關鍵）| 下拉選，**預設＝`Context-specific 差異 (Stim−Rest)`**；其餘：`效應廣度`(次選) / `可成藥性` / `遺傳約束 LOEUF` | 切換即重算排名 |
| 篩選側欄 | 條件、通過門檻 (gate)、有無 off-target、模組(TCR/SAGA/Mediator)、可成藥性 | 多選 filter |
| **Caveat 橫幅**（常駐）| 「⚠ 依『效應廣度』排名時，SAGA/Mediator/核糖體等泛效應基因會被推高——這是效應廣度≠治療相關性。建議用 Context-specific 指標。」 | 可收合但預設展開 |
| 篩選漏斗小圖 | QC funnel (C3)：33,983 → 2,131 profiles (1,235 targets)，顯示目前 filter 篩掉多少 | hover 看各階段數字 |

> **PM 註：** 排名指標切換是 Entry A 的靈魂。複查報告 §3.1 指出目前「效應廣度」排名被泛效應污染；UI 必須讓研究者**一鍵切到 context-specific 指標**，並在旁邊看到排名如何改變。這把「找效應大的基因」升級成「找 T 細胞特異調控子」。

### L2 — Target Dossier（標的證據卡）

**回答的問題：** 「這一個標的，所有證據攤開給我看。」

單一標的的整合卡，分證據軸（每軸可收合，預設顯示摘要 + 信心燈號）：

```
┌─ CD3E ────────────────────────── Grade 3 · Readiness R3 ─┐
│ [證據強度雷達圖 V4]        [三條件效應 dot plot C1]        │
├──────────────────────────────────────────────────────────┤
│ ▸ 功能證據   下游 DE 廣度: Rest 4 → Stim8hr 5.7k (context!)│
│ ▸ 通路模組   TCR 近端訊號 (STRING: CD3D/CD3G/CD247 0.999) │
│ ▸ 可成藥性   Open Targets tractability 10/28 · SM+AB      │
│ ▸ 遺傳約束   gnomAD LOEUF 0.70 · pLI 0.05 (容忍 LoF)       │
│ ▸ 臨床先例   22 藥 · 9 個 T-cell engager (BLINATUMOMAB…)  │
│ ▸ 安全訊號   Open Targets safety liabilities: 無          │
│ ▸ ADC 可及性 表面蛋白 ✓ · extracellular 104 aa            │
│ ▸ 文獻       10 篇 PubMed (fetched 2026-07-08)            │
└──────────────────────────────────────────────────────────┘
```

- 每軸右側一個**信心燈號**（綠/黃/灰＝強/中/無證據），讓研究者三秒抓到證據分佈。
- **每個數字可點** → L3 看原始 API 回應 + 時間戳 + source_version。
- 頂部 Grade(1-4) 與 Readiness(R0-R5) 是彙總分級，hover 顯示計分邏輯。

### L3 — Source Drill-down（來源鑽取）

點任一證據 → 側抽屜(drawer)展開原始資料：原始 GraphQL/REST 回應摘要、擷取時間、來源版本、外部連結。**這是「證據可追溯」的落點**——研究者要能自己核對，不是只看我們的彙總。

### 1.4 Entry A 的視覺化對應（用既有 53 張圖庫）

| 用途 | 圖 | 位置 |
|---|---|---|
| 標的排名 | R1 Lollipop / R2 Diverging bar | L1 主圖 |
| 三合一(效應/廣度/細胞數) | C1 3D dot plot | L2 卡頂 |
| 證據多維度 | V4 Radar | L2 卡頂 |
| 通路模組歸屬 | N1 STRING 網路 / N7 UpSet | L2 通路軸 |
| context-specificity | E 基因×條件 heatmap（Figure 1e） | L2 功能軸 |
| 篩選流程 | C3 QC funnel / H4 Sankey | L1 側 |
| 成藥性/安全散點 | R5 Bubble / R9 MA | L2 展開 |

## 2. ENTRY B — 臨床醫師：Patient Risk & Evidence Lookup

**核心任務：** 「病人已知某疾病/帶某變異，用他自己的資料，這條基因通路對他是風險還是治療機會？證據多強？」

> ## ⚠️ 責任邊界（產品層強制，不可繞過）
> **本入口為研究用證據查詢與假設生成工具，非診斷或處方系統。所有輸出為族群層級的證據強度與假設，不構成對特定病人的醫療建議。任何治療決策必須由具完整病人脈絡的合格醫療專業人員做出。**
> - **醫療專業身分驗證閘門（gated）**：Entry B 與 Entry A 不共用登入；進入前需通過醫療專業身分驗證，**驗證未過不得上傳任何病人資料**。此 disclaimer 於驗證後首次進入需確認，並在每份報告頁首常駐。
> - **病人資料＝純 session**：只在本次瀏覽階段記憶體處理，關閉/逾時即清除，**不落地、不外傳、不訓練模型**；無「儲存病例」功能。UI 明示「資料僅於本次 session 處理，不儲存」。
> - 輸出用語一律「族群層假設 / 證據強度」，**禁用**「建議用藥 / 診斷 / 預後」等處方性字眼。

### 2.1 畫面流

```
[B1 資料上傳] ─▶ [B2 風險-證據對照] ─▶ [B3 通路/族群脈絡] ─▶ [B4 可下載報告]
  Upload           Risk × Evidence         Context               Report
```

### B1 — Upload（病人資料上傳）

**回答的問題：** 「把我手上的病人資料接進來。」

- 支援：基因清單 / 變異清單 (VCF-like) / 表現量表 / 已知疾病代碼(如 ICD/EFO)。
- 上傳後即時 QC：辨識出幾個基因、幾個對得上平台的 1,235 標的 / 11,526 全體。
- **明示邊界卡**：「已辨識 N 個基因，其中 M 個有 GWT T 細胞擾動證據。以下為這些基因的**族群層證據**，非病人層預測。」

### B2 — Risk × Evidence（風險-證據對照，Entry B 主畫面）

**回答的問題：** 「病人這些基因，哪些帶已知風險訊號？證據多強？」

雙軸對照矩陣（低認知負荷的核心設計）：

```
        證據強度 (evidence strength) ──▶
  風 高 │  ● PLCG1        │              │
  險 中 │  (Angioedema    │  ● MED12     │
  訊 低 │   safety +      │  (LOEUF 0.095│
  號    │   LOEUF)        │   極約束)    │
        └─────────────────┴──────────────┘
              可作用            僅假設
```

- 每個病人基因一個點：**y=風險訊號強度**（gnomAD 約束 + Open Targets safety liability + GTEx 關鍵組織表現）、**x=證據充分度**（有幾條獨立來源支持）。
- 右上＝高風險且證據足；左上＝高風險但證據薄（需謹慎）。
- 點任一 → 展開該基因對這位病人的**族群層證據卡**（複用 Entry A 的 L2 引擎，但語氣改成風險脈絡）。
- **安全訊號優先醒目**：如 PLCG1→Angioedema、MED12 極 loss-intolerant，用警示色標出。

### B3 — Context（通路與族群脈絡）

- **通路脈絡**：病人的基因落在哪些通路模組(Reactome/STRING)？若集中在 TCR 訊號，提示這是 T 細胞活化相關。
- **族群假設**（UK Biobank）：如 PLCG1 淋巴球數 post_mean +0.29——**明示為族群統計關聯，非此病人預測**。
- **疾病-藥物證據匹配**（demo）：病人疾病 × 已知標的藥物的臨床試驗數（如 basiliximab × 腎移植 111 試驗）——標明「文獻證據，非處方」。

### B4 — Report（可下載報告）

- 一頁式 PDF，採**可帶機構 logo 的免責頁範本**（供醫師在其機構實際使用）：病人基因 → 風險-證據對照 → 通路脈絡 → **全文 disclaimer**。
- **範本結構**：頁首（機構 logo 欄位 + 常駐 disclaimer 一行）→ 正文（每個結論附證據來源與時間戳）→ 專屬免責頁（完整 disclaimer 段 + 產生日期 + 資料來源版本清單 + 「純 session 處理、未儲存」聲明）。
- 頁尾常駐：「本報告為研究用證據彙整，供醫療專業人員參考，非診斷或處方。」

### 2.6 Entry B 的視覺化對應

| 用途 | 圖 | 位置 |
|---|---|---|
| 風險×證據對照 | 自訂 2D scatter（風險 vs 證據強度） | B2 主圖 |
| 上傳資料 QC | C3 funnel（辨識/對得上/有證據） | B1 |
| 通路歸屬 | N1 STRING / N3 Venn | B3 |
| 遺傳約束 | R7 誤差棒 / gnomAD LOEUF 排序 | B2 展開 |
| 組織表現(脫靶風險) | M6 條件×指標 heatmap / GTEx bar | B2 展開 |

## 3. 互動儀表板規格（兩入口共用元件）

**設計原則：toggle 不重畫整頁，只重算受影響面板（延續 STAGE2 互動規格）。**

| 元件 | 行為 | 適用 |
|---|---|---|
| **條件切換** Rest/Stim8hr/Stim48hr | 全域 toggle，所有圖同步 | A + B |
| **排名指標下拉** | 廣度/context-specific/成藥性/約束 | A (L1) |
| **證據來源開關** | 勾選要納入的證據軸（OT/gnomAD/CT/pathway…） | A + B |
| **門檻滑桿** | n_cells / DE genes / significance 即時篩 | A (L1) |
| **模組 filter** | TCR / SAGA / Mediator | A + B |
| **鑽取抽屜** | 點數字 → 側邊展開原始來源 | A + B |
| **匯出** | PNG(圖) / CSV(表) / PDF(報告) | A + B |

**低認知負荷落實：**
- 首屏只給一張主圖 + 一個主 filter；其餘收合。
- 顏色語意全站一致：三條件 藍/橙/綠；風險 = 警示紅；證據足 = 綠燈。
- 每張圖標題是結論句，讀者不用解讀。

## 5. MVP 建置優先序（PM 排程）

1. **P0** — Entry A 的 L1 Rank Board + 排名指標切換（含 context-specific）+ Caveat 橫幅。這是最能展示核心價值、也最能修正 §3.1 排名問題的一屏。
2. **P0** — Entry B 的 disclaimer 閘門 + B1 上傳 + B2 風險×證據對照。責任邊界必須在任何病人資料進入前就位。
3. **P1** — L2 Target Dossier 證據卡（兩入口共用引擎）+ L3 來源鑽取。
4. **P1** — 互動儀表板共用元件（條件切換、證據開關、鑽取抽屜）。
5. **P2** — B4 報告匯出、族群假設脈絡、疾病-藥物匹配 demo。
6. **P2** — 進階圖（STRING 網路、UpSet、Sankey）嵌入卡片。

## 6. 已拍板決定（PM 定案 2026-07-08）

1. **Entry B 病人資料儲存 = 純 session。** 資料只在本次瀏覽階段記憶體處理，關閉/逾時即清除，不落地、不外傳、不訓練模型。UI 明示「資料僅於本次 session 處理，不儲存」；無「儲存病例」功能。→ 影響 §2、§B1。
2. **排名預設指標 = Context-specific 差異 (Stim−Rest)。** 首屏預設即用此指標，避免泛效應誤導；效應廣度改為可切換的次選項（並保留 Caveat 橫幅說明差異）。→ 影響 §1 L1 Rank Board。
3. **兩入口不共用登入；Entry B 需醫療專業身分驗證。** Entry A 一般登入即可；Entry B 進入前需通過醫療專業身分驗證（gated），驗證未過不得上傳任何病人資料。→ 影響 §0 IA、§2 責任邊界。
4. **需機構免責範本。** B4 報告 PDF 採可帶機構 logo 的免責頁範本，頁首/頁尾常駐 disclaimer，供醫師在其機構實際使用。→ 影響 §B4。
